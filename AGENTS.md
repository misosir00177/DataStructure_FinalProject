## Project Overview
本專案是以「二手物品以物易物、無金融交易」為主題的 app 前端展示 demo。
目標是在瀏覽器中可以打開的靜態網站，內容展示：
- 模擬拍攝物品與 AI 圖像快速分析標籤功能
- 物品篩選與過濾條件設定
- 類似交友軟體的快節奏滑動配對介面（左滑忽略、右滑喜歡）
- 配對成功（Match）畫面與交換約定介面（純物品交換，無金融交易）
- 資料結構如何被使用
本專案主要用於課堂展示與期末報告，不是實際平台，不需要架設真實的後端，採用假的後端。

## Tech Stack
請只使用：
- HTML
- CSS
- JavaScript
- 不使用後端
- 不使用資料庫
- 使用假的登入系統
最終作品必須可以直接用 `index.html` 開啟。

## File Structure
請維持以下檔案結構：
├── index.html
├── style.css
├── script.js
├── data/    # 任何其他資料都放置於此
├── AGENT.md
├── PRD.md
├── USER_STORY.md
└── README.md

## Must Have
- 共同功能
  - 假的登入與個人主頁介面
  - 物品過濾與標籤篩選器
  - 雙向配對成功與約定交換介面（需強調無金錢交易）
- 核心操作流程
  - 模擬相機拍攝與假 AI 圖像辨識：點擊上傳後，系統模擬分析並自動生成相近標籤（如：3C、書籍、九成新）。
  - 快節奏卡片滑動介面：實現向右滑（想交換）、向左滑（跳過）的互動效果。
- 使用 Queue (佇列) 或 Stack (堆疊) 來管理滑動卡片，或使用 Hash Map / Set 進行標籤比對配對的說明。
- 適合簡報展示的畫面

## Do Not Do
- 不要做太複雜的系統
- 不要加入真實 API（請勿串接真實的 AI 圖像辨識 API，用 `setTimeout` 模擬即可）
- 不要重寫全部程式
- 不要做需要安裝才能跑的功能
- 無需真實的帳號密碼驗證

## 驗收標準 (重要)
### 1. 基本執行
- 使用者可以直接打開 `index.html` 看到完整畫面
- 不需要安裝額外套件
- 假的登入畫面
- 虛假後端
- 不需要 API key
- 頁面載入後不能出現明顯錯誤畫面
- 瀏覽器 console 不應該有會影響功能的 JavaScript error

### 2. Demo 流程
使用者必須能完成一個完整展示流程：
1. 使用者以假帳號登入，進入上傳物品介面。
2. 模擬拍攝自己的二手物品，系統透過假 AI 跑出分析動畫並自動標註「物品標籤」。
3. 進入首頁設定「篩選器」，避開自己不想要的物品類別。
4. 系統利用 Queue/Array 載入過濾後的二手物品卡片堆。
5. 使用者進行滑動操作（向左滑掉不喜歡的，向右滑動喜歡的）。
6. 滑動到特定卡片時，系統觸發「配對成功 (It's a Match!)」的雙向媒合畫面。
7. 進入模擬聊天室或約定介面，確認以物易物的交貨地點。
如果是互動式作品，至少要有一條「保證可成功展示」的 demo path

### 3. 核心功能
- 至少完成 MVP 中定義的核心功能。
- 核心功能不能只是靜態文字描述，必須有可操作或可觀察的結果。
- 每個按鈕或主要互動元件都應該有明確作用。
- 不應存在點了沒有反應、但看起來像主要功能的按鈕。
- 如果功能尚未完成，請不要放在主要畫面上誤導使用者。

完成時必須能回答：
- 使用了哪一種資料結構？
- 資料是怎麼存的？
- 為什麼這裡適合用這種資料結構？
- 這個資料結構如何影響功能運作？

### 4. 檔案與程式碼
- 主要檔案命名清楚，例如 `index.html`, `style.css`, `script.js`。
- 不要把大量 JavaScript 全部塞在 HTML 裡。
- 不要留下大量沒用到的程式碼。
- 不要留下測試用假按鈕、假頁面或無意義文字。
- 函式名稱應該能看出用途。
- 如果有資料檔，資料結構要清楚，不要全部寫成難以閱讀的一大坨字串。

### 5. 穩定性
- 重新整理頁面後仍可以正常使用。
- 操作順序稍微不同時，不應直接壞掉。
- 沒有選資料就按按鈕時，系統應該給出提示，而不是報錯。
- 如果輸入錯誤，畫面應該有清楚回饋。
- Demo 前應至少完整跑過 3 次主要流程。

### 6. 範圍控制
以下內容不是必要項目，不應該影響 MVP 完成：
- 後端資料庫
- 真實 AI 圖像辨識模型或 API
- 真實物流與通知系統
- 大量頁面
- 過度精緻但無法操作的動畫

### 7. 最終完成定義
當以下條件都達成時，才算完成：
- 可以正常開啟作品。
- 可以完成主要 demo 流程。
- 有清楚的輸入、處理、輸出。
- 至少一個資料結構有被實際用在功能中。
- 畫面適合展示。
- 團隊能說明作品邏輯。
- 沒有明顯會讓 demo 中斷的錯誤。
